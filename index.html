1. Introduction

1.1 Background

With the rapid growth of digital systems and online services, organisations generate massive volumes of security logs from servers, firewalls, applications and network devices. Manually inspecting these logs is inefficient and easy to get wrong. Meanwhile, cyber‑attacks such as unauthorised access, brute‑force login attempts and credential stuffing continue to increase.

1.2 Problem Statement

Traditional log‑analysis tools struggle when log volumes reach gigabytes or terabytes per day. Security teams need a scalable way to:

  • Filter and aggregate millions of log entries efficiently.
  • Detect abnormal or suspicious behaviour in near real time.
  • Generate clear reports summarising potential threats.

1.3 Objectives

The main objectives of this project are:

  • To apply Big Data concepts using Apache Spark for cybersecurity analytics.
  • To analyse large‑scale log data and detect suspicious activities automatically.
  • To implement a scalable data‑processing pipeline on Google Colab.
  • To demonstrate how Big Data tools can strengthen practical cybersecurity monitoring.
Key Idea

Instead of manually reading log files, we let Apache Spark do the heavy work: loading, filtering, grouping and highlighting security anomalies at scale.

2. Big Data Tools & Environment

2.1 Technology Stack

  • Apache Spark (PySpark): distributed processing engine for loading and analysing log data.
  • Google Colab: cloud‑hosted Jupyter environment used to run Spark code without local setup.
  • Local / Cloud Storage (CSV files): stores raw synthetic logs and exported suspicious‑IP reports.
  • Python Ecosystem: NumPy and pandas for data generation and simple plotting.

2.2 Spark on Google Colab

Google Colab provides free compute resources and an interactive notebook interface, which is ideal for experimenting with Big Data frameworks. In this project, Spark is installed and configured inside Colab, allowing us to:

  • Create a SparkSession directly in the notebook.
  • Read and write CSV files stored in Colab’s virtual file system.
  • Visualise results with Python plotting libraries.

3. Log Dataset Design

3.1 Schema & Fields

To avoid privacy issues, the project uses synthetic security logs generated inside the notebook. Each log entry simulates an authentication or access event and contains:

  • timestamp – date and time of the request.
  • ip_address – source IPv4 address (normal or intentionally suspicious).
  • username – user or service account attempting access.
  • endpoint – resource being accessed (e.g. /login, /admin).
  • status – SUCCESS or FAIL.
  • country – rough geographic origin for additional context.

3.2 Volume & Storage

The notebook generates around 8,000 log entries spread across multiple days. Although relatively small compared to enterprise systems, the structure allows the same Spark pipeline to be applied to millions of rows with only configuration changes.

Generated logs are first built in a pandas DataFrame and then saved as cyber_logs.csv, which Spark reads into a distributed DataFrame for further analysis.

4. Implementation Pipeline

4.1 Step‑by‑Step Flow

Install PySpark and create a SparkSession inside Google Colab.

Generate synthetic security logs with mixed normal and suspicious activity.

Save the logs as CSV and load them into a Spark DataFrame.

Clean and enrich the data (e.g., convert timestamps, extract hour of day).

Apply filters, group‑by and aggregations to compute security metrics.

Apply rule‑based threat detection to flag suspicious IP addresses and time periods.

Export the list of suspicious IPs to CSV for reporting or further visualisation.

5. Threat Detection Analytics

5.1 Detection Rules Implemented

Three primary rule‑based analytics methods are implemented using Spark transformations:

  • Rule 1 – Brute‑Force Login Detection: IP addresses with more than a chosen threshold (e.g. 25 failed logins) are flagged as potential brute‑force attackers.
  • Rule 2 – Credential Stuffing / User Enumeration: For each IP, the number of distinct usernames with failed logins is counted. IPs failing against many different usernames are considered suspicious.
  • Rule 3 – Abnormal Hourly Activity: Total log events per hour are calculated and compared to the average. Hours with counts above mean + 2 × standard deviation are highlighted as unusual.
Spark for Security

Because these rules are implemented with groupBy, agg and filter over a Spark DataFrame, they naturally scale to millions of log entries spread across a cluster.

5.2 Key Metrics & Outputs

  • Top IPs by number of failed login attempts.
  • IPs associated with many different failed usernames.
  • Distribution of log volume per hour and detection of “noisy” hours.
  • Combined suspicious‑IP table with reasons and counts, saved as CSV.

These metrics can be easily exported from Spark to tools such as Excel, Power BI or a SIEM dashboard for further inspection by security analysts.

7. Conclusion & Future Work

7.1 Conclusion

This Big Data project shows how Apache Spark can be used as a powerful engine for cybersecurity analytics. By generating and processing large‑scale synthetic logs, we were able to detect brute‑force attempts, credential‑stuffing behaviour and abnormal traffic hours using simple, explainable rules.

Even though the logs in this project are simulated, the same pipeline can be applied to real enterprise data sources such as web server logs, VPN logs or SIEM exports, making it a valuable starting point for building more advanced threat‑detection systems.

7.2 Future Improvements

  • Integrate real‑time log ingestion using Spark Streaming or Apache Kafka.
  • Deploy the Spark job to cloud platforms like AWS EMR or Azure Databricks.
  • Incorporate machine‑learning models (e.g. anomaly detection, classification) for smarter alerts.
  • Connect the output to a dashboard or SIEM tool for continuous monitoring.

Overall, the work reinforces a key lesson of Big Data and cybersecurity: large log volumes are only useful when combined with scalable analytics and clear, actionable insights.
